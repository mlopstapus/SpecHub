# Research: Tenant Isolation Tests & RLS

## Methodology

Rather than treating "audit every service-layer query" (FR-004) as a formality, every `identity-access` application-layer function and repo query was read in full (`src/bcs/identity-access/{application,infrastructure}/*.ts`, excluding `.test.ts`). Findings below are grounded in exact file/line references, not assumptions. The single biggest discovery: **RLS on `users`/`api_keys`/`invitations`/`organizations`, if enabled exactly as tenet M2 describes with no further design, would break four already-shipped flows** that must resolve an identity with *no organization context yet* (login by email, session auth by JWT subject, API-key auth by hash, invitation acceptance by token). Resolving that is the majority of this feature's real design work — the migration SQL itself is comparatively small.

## 1. RLS policy shape (FR-001/FR-002)

**Decision**: Standard session-variable policy per `context/database-conventions.md`, applied to `teams`, `users`, `invitations`, `api_keys` (`organization_id = current_setting('app.current_org_id')::uuid`) and to `organizations` itself (`id = current_setting('app.current_org_id')::uuid`, since it has no `organization_id` column — FR-002, this feature's `/speckit-clarify` answer). Every policy is scoped `TO skillcanon_app` explicitly (not left as the default `PUBLIC`), so a second, differently-scoped role can coexist on the same table (see §2).

**Rationale**: Exact pattern already proven correct in this codebase — `tenant-context.test.ts` validates the identical shape (`ENABLE ROW LEVEL SECURITY` + `USING (organization_id = current_setting(...)::uuid)`, set via `withTenantContext`'s `set_config(..., true)`) against a throwaway smoke-test table in `002-drizzle-shared-db-kernel`. This feature applies the same proven mechanism to the five real tables.

**Alternatives considered**: Role-per-tenant, JWT-claim-reading policy functions — both already rejected by `context/database-conventions.md` for the reasons stated there; not re-litigated here.

## 2. The credential-resolution problem and the `skillcanon_auth` role

**Finding**: Four existing, already-shipped functions each perform a lookup that is *legitimately* cross-organization, because they resolve "who is this" before any org is known:

| Function | File | Cross-org read |
|---|---|---|
| `login` | `application/login.ts:37` | `findByEmail` — searches `users` across **all** orgs (email is only unique per-org, PDR-003; already documented in `008-jwt-session-auth`'s research.md §8) |
| `authenticateSession` | `application/authenticate-session.ts:42` → `get-user.ts:15` | `getUser`'s `findById` — JWT carries only `sub` (user id), no org claim (`CONTRACT.md`: `sub`/`role`/`exp` only) |
| `authenticateApiKey` | `application/authenticate-api-key.ts:25,33` | `findByHash` (api_keys) then `findUserById` (users) — a raw bearer key carries no org either |
| `acceptInvitation` | `application/accept-invitation.ts:49` | `findByToken` — an invitation link is emailed to someone who isn't a member (and thus has no org context) yet |

Once RLS is enabled on `users`/`api_keys`/`invitations` with a policy scoped only to `skillcanon_app`, every one of these reads returns **zero rows** when run through the ordinary app connection with no `app.current_org_id` set — `set_config`'s counterpart, `current_setting(...)` with no default, actually *throws* rather than returning empty (proven today by `tenant-context.test.ts`'s "querying with no tenant context established is denied outright" case). Left unfixed, this feature would silently break login, session auth, API-key auth, and invitation acceptance the moment its own migration lands.

A fifth, related case: `createOrganization`'s self-hosted single-org guard (`organizations-repo.count()`, `application/create-organization.ts:51`) needs the **true global count** of organizations to enforce "at most one, in self-hosted mode" — impossible under an `id = current_org_id` policy, which can only ever show 0 or 1 row (whichever matches the session value, if any).

**Decision**: A second Postgres role, `skillcanon_auth`, provisioned in the same idempotent `DO $$ ... $$` block style as `skillcanon_app` (`drizzle/migrations/0000_create_schemas.sql`). It gets one additional permissive policy per table — scoped `TO skillcanon_auth`, `USING (true) WITH CHECK (true)`, covering all commands — on all five `identity_access` tables (`organizations`, `teams`, `users`, `invitations`, `api_keys`), plus the matching `GRANT SELECT, INSERT, UPDATE ON ... TO skillcanon_auth` (no `DELETE` — nothing in this bounded context's current flows needs it via this role).

This is deliberately **narrower than the migration/owner role** in two ways that matter for blast radius: scoped to `identity_access` (plus one narrow addition — `SELECT, INSERT` on `audit.audit_events`, since `login`/`acceptInvitation` audit-log within the same transaction as their pre-auth reads. Discovered in two stages against the first draft of this migration: the existing test suite's audit-event assertions first failed with `permission denied for schema audit` (missing `USAGE`), and after adding the schema grant plus an `INSERT`-only table grant, failed again with `permission denied for table audit_events` — because `audit-compliance`'s `record()`/`insert()` uses `.returning()`, and Postgres requires `SELECT` privilege on any column an `INSERT ... RETURNING` clause returns, not just `INSERT`), and excludes `DELETE`/`UPDATE` on `audit.audit_events` and `DELETE` everywhere else in `identity_access`. It is **broader than a SELECT-only design** (an earlier draft of this research considered SELECT-only, but `acceptInvitation`'s `markAccepted` UPDATE, `insertValidatedUser`'s INSERT, and `authenticateApiKey`'s `updateLastUsedAt` UPDATE all need to run in the same trusted, no-context-yet path) — see "Alternatives considered" below for why a stricter split was rejected.

**Consequence for existing code**: because every one of `login`, `authenticateSession`, `authenticateApiKey`, `acceptInvitation`, `createOrganization`/`bootstrapOrganization` already receives its DB handle as a plain parameter (`db: PostgresJsDatabase<...>`, no internal singleton import), **none of their function bodies need to change**. Only the caller's choice of *which role's connection* to pass changes — today, only tests; in production, whichever future `007-distribution` route handler invokes them. This is documented per-function in data-model.md so `007-distribution` doesn't have to rediscover it.

**`client.ts`/`test-helpers.ts` additions**: `shared/db/client.ts` gains a second lazily-initialized export, `authDb`, connected via a new `AUTH_DATABASE_URL` env var (mirrors `db`'s existing pattern exactly, including the placeholder/missing-value guard). `shared/db/test-helpers.ts`'s `TestDb` interface gains an `authDb` field (role: `skillcanon_auth`, same throwaway-container credentials pattern as `appDb`).

**Alternatives considered**:
- *Reuse the existing migration/owner role* for these four flows. Rejected: the owner role has unrestricted DDL+DML across all seven schemas; using it on every login request needlessly widens a bug or injection's blast radius from "this one bounded context, no deletes" to "everything." `context/database-conventions.md`'s FR-010 framing ("a role distinct from the ordinary application runtime role") is satisfied more defensibly by a narrow, purpose-built role.
- *SELECT-only `skillcanon_auth`, with the actual writes (audit record, `markAccepted`, `insertValidatedUser`, `updateLastUsedAt`) split into a second, properly-`withTenantContext`-scoped call once the org is known.* Considered in depth (would require `acceptInvitation`/`authenticateApiKey` to take two DB parameters instead of one). Rejected as unnecessary complexity: it doesn't shrink the role's schema-level reach at all (still `identity_access`-only), only trades a same-transaction write for a second connection/transaction — and for `createOrganization` specifically, splitting the advisory-lock-guarded count-then-insert sequence across two connections would have reintroduced the exact TOCTOU race the lock exists to prevent (the lock is transaction-scoped; the insert would need to happen on the *same* still-open transaction that holds it).
- *A single, more permissive `USING(true)` policy applied to `skillcanon_app` itself for these cases* — rejected outright: that would silently disable RLS for every ordinary request too, not just the four legitimate no-context flows.

## 3. `getUser`/`getTeamChain`/`getOrganization` — closing the app-layer (M1) gap

**Finding**: Unlike every mutation in this bounded context (`updateUser`, `deactivateUser`, `revokeApiKey`, `revokeInvitation`, `createTeam`, `reparentTeam`, all of which fetch via an org-scoped repo query or explicitly compare `.organizationId` before acting — see `users-repo.findByOrgAndId`, `invitations-repo.findByOrgAndId`, `api-keys-repo.findByOrgAndId`, and app-layer comparisons in `update-user.ts:58`, `create-team.ts:18`, `reparent-team.ts:35`), three read-only, cross-BC-exposed functions have **no app-layer org check at all**:

- `getUser(db, userId)` (`application/get-user.ts:15`) — `users-repo.findById`, no scoping.
- `getTeamChain(db, teamId)` (`application/get-team-chain.ts:32`) — `teams-repo.findById`, no scoping, walked repeatedly.
- `teams-repo.ts` has no `findByOrgAndId` at all today (unlike `users-repo`/`invitations-repo`/`api-keys-repo`), confirming this wasn't a deliberate omission matching an established pattern — the pattern exists everywhere else in this BC and simply wasn't applied here yet.

`CONTRACT.md` documents both as taking a bare id, consumed by "All contexts" (`getUser`) and "Governance" (`getTeamChain`) — i.e., these are real cross-BC contract calls, not just internal helpers.

However, `getUser` has one legitimate reason to stay unscoped in one specific call path: `authenticateSession` (`application/authenticate-session.ts:42`) calls it with only a JWT subject, before any org is known — exactly the class of problem in §2.

**Decision**:
- `getUser(db, userId, organizationId?)` — `organizationId` becomes an **optional** third parameter. When provided, the function uses `users-repo.findByOrgAndId` (already exists) and throws the same not-found-equivalent error for a wrong-org id as for a nonexistent one (matching this BC's established denial semantics — see §5). When omitted, it falls back to today's unscoped `findById` — this path is now documented as intentionally narrow, used only by `authenticateSession`'s pre-auth resolution (called via `authDb`, per §2). Any future "All contexts" consumer that already has its own org context (the documented, common case) is expected to pass it.
- `getTeamChain(db, organizationId, teamId)` — `organizationId` becomes a **mandatory** new parameter (no legitimate unscoped consumer: Governance, per `CONTRACT.md`, always already has its own org context before asking for a chain). Add `teams-repo.findByOrgAndId(tx, organizationId, id)`, mirroring the other three repos, and use it for the **starting** lookup only. Subsequent ancestor-walk steps keep using plain `findById` — safe without re-checking, because `createTeam`/`reparentTeam` already guarantee (via `CrossOrgReparentError`) that a team's parent always shares its organization, so once the starting team's org is confirmed, every ancestor transitively shares it too.
- `getOrganization(db, organizationId)` — **no signature change.** Unlike the two above, there is no second identity to compare against: the single parameter *is* the org being fetched, and the only legitimate use is "fetch my own, already-established org." Once RLS lands (`id = current_org_id`), a caller whose connection is properly `withTenantContext`-scoped gets this as a real, working backstop; there is no additional app-layer check to add without a redundant second "expected id" parameter that would always have to equal the first. Documented here so this isn't mistaken for an overlooked gap during future review.

**Consequence**: `CONTRACT.md`'s two listed signatures (`getUser(userId)`, `getTeamChain(teamId)`) are updated to match. This is a breaking change to a documented contract, judged safe now because no real consumer exists yet outside this epic's own tests (Governance, the only "All contexts"/"Governance" consumer named, doesn't exist as a bounded context yet).

## 4. Existing test suite: the real migration cost

**Finding**: Zero files under `src/bcs/identity-access` call `withTenantContext` today (`grep -rl withTenantContext src/bcs/identity-access` → no matches). 28 test files call functions directly against `testDb.appDb`, uniformly with no tenant context established. Once RLS lands, every one of these breaks — not "returns empty," but throws (`current_setting` with no default and no value set raises an error, proven by `tenant-context.test.ts`).

**Decision — two-part fix, applied per test file**:
1. **Fixture/setup calls** (creating the organization, team(s), user(s), invitation(s), or API key(s) a test needs as scaffolding, not what's under test) switch from `testDb.appDb` to `testDb.authDb`. This is exactly the role built for "no tenant context established yet" (§2), and it is the correctly-scoped choice for fixture setup specifically because it's confined to `identity_access` (unlike the broader owner role) while still being unrestricted-by-org.
2. **The function-under-test call** (and any assertion query reading back its effect), for every function that represents "an authenticated user acting within their own org" — the overwhelming majority: `createTeam`, `updateTeam`, `reparentTeam`, `insertTeamBetween`, `createUser`, `updateUser`, `deactivateUser`, `inviteUser`, `revokeInvitation`, `createApiKey`, `revokeApiKey`, `listApiKeys`, `listUsers`, `listInvitations`, `listSubTeams`, `getUser` (with its new org param), `getTeamChain`, `getOrganization` — is wrapped in `withTenantContext(testDb.appDb, knownOrgId, tx => ...)` instead of a bare `testDb.appDb.transaction(...)` or direct `testDb.appDb` call. This is not just mechanical churn: it's what makes each of these tests exercise the *real* production shape (RLS-subject connection, context established exactly as a real request would) rather than merely re-testing app-layer logic that happened to also pass before RLS existed.
3. The four no-context flows (`login`, `authenticateSession`, `authenticateApiKey`, `acceptInvitation`) and the bootstrap flow (`createOrganization`/`bootstrapOrganization`) keep calling through `testDb.authDb` end-to-end in their own tests too — this is their real, intended production shape, not a test-only convenience, so it's valuable to prove it works under actual RLS rather than only under `ownerDb`'s full bypass.

**Rationale for not using `ownerDb` for any of this**: `ownerDb` bypasses RLS on *every* schema, not just `identity_access` — reaching for it here would prove nothing about whether this feature's actual RLS policies are correctly shaped, and would silently hide a regression the same way not testing RLS at all would.

## 5. Cross-tenant-denial test helper shape (FR-005/FR-009)

**Decision**: `src/shared/testing/tenant-isolation.ts`, exporting `assertCrossTenantDenied` matching the shape already sketched in `context/testing-strategy.md`'s "M3 cross-tenant-denial pattern" section (this feature is that section's own implementation, not a new design):

```ts
export async function assertCrossTenantDenied(opts: {
  actingAsOrg: string;
  resourceOwnedByOrg: string;
  fetchResourceById: (id: string) => Promise<unknown>;
  resourceId: string;
}): Promise<void>
```

Asserts the fetch throws or resolves to nothing — never that it merely omits the resource from some other, list-shaped result — matching M3's own stated rationale. Documented with a runnable usage example in `context/testing-strategy.md` (FR-009) rather than only in this feature's own test files, so a later epic's tenant-isolation-tests feature can find it without reading this feature's implementation.

**Two distinct denial mechanisms, one helper**: this feature needs the helper to prove *both* (a) the app-layer function denies cross-org access (calling, e.g., `getUser(tx, otherOrgUserId, myOrgId)` and expecting a throw) and (b) RLS alone denies it even with no app-layer check in the way (calling the underlying Drizzle query directly — e.g. `tx.select().from(users).where(eq(users.id, otherOrgUserId))`, with `tx` from `withTenantContext(appDb, myOrgId, ...)` — and expecting zero rows). `fetchResourceById` is intentionally just `(id) => Promise<unknown>`, so both call shapes fit through the same helper without a variant per mechanism.

## 6a. `logout` — a fifth no-context flow, found during test migration

While migrating the existing test suite (§4), `logout.test.ts` surfaced a fifth function needing `authDb`: `logout(db, userId)` internally calls `getUser(db, userId)` with no `organizationId` (it only ever receives a bare `userId`, the same shape `authenticateSession` resolves from a JWT subject). This wasn't caught during the initial planning-phase audit (§3) because that audit focused on `getUser`'s *callers with an org context available*, not on `logout`'s own indirect, contextless use of it. Added to `bcs/identity-access/CONTRACT.md`'s `authDb`-required list alongside `login`/`authenticateSession`/`authenticateApiKey`/`acceptInvitation`/`createOrganization`/`bootstrapOrganization`. This is itself a small, concrete instance of exactly what User Story 3 asks for: the audit isn't complete until every real call site is checked, not just the ones anticipated up front.

## 6b. `reparentTeam`'s `CrossOrgReparentError` branch — now unreachable via a properly-scoped connection

Another finding from test migration: `reparent-team.test.ts`'s "rejects reparenting under a team from a different organization" test originally asserted `CrossOrgReparentError`. Under real RLS, a session scoped to organization A cannot see organization B's team row at all — `reparentTeam`'s `findById(tx, newParentId)` returns `undefined` (RLS hides it, not just "genuinely doesn't exist"), so its earlier `if (!newParent) throw new Error(...)` branch fires instead of the later, more specific `if (newParent.organizationId !== team.organizationId) throw new CrossOrgReparentError()` branch. This differs from `createTeam`'s and `insertValidatedUser`'s equivalent checks, which use a single unified `if (!row || row.organizationId !== expected) throw SameError()` — those still throw their original, specific error under RLS, since both "not found" and "wrong org" already led to the same branch even before RLS existed.

This is not a regression: the rejection still happens, and FR-008 already establishes that this bounded context treats "wrong org" as indistinguishable from "not found" by design. It does mean `CrossOrgReparentError`'s second branch in `reparentTeam` is dead code when reached through an RLS-scoped connection (the *only* way it fires now is if RLS were somehow bypassed or misconfigured — ironically making it a real, if accidental, second-layer check). Recorded here for T044's audit pass; no code change made, since collapsing the two branches would be a larger, unrequested refactor of already-shipped `006-team-hierarchy` code for a purely cosmetic error-type distinction that FR-008 says shouldn't be observable anyway.

## 3a. Final audit pass (2026-07-23, T044) — one more real gap found and fixed

A final re-review of every `identity-access` repo function and its application-layer callers, performed after the existing test suite was fully migrated to real RLS (so any gap would surface as a test failure or an obviously-wrong silent success), found one more genuine M1 gap beyond `getUser`/`getTeamChain`:

- **`updateTeam(tx, teamId, fields)`** had no `organizationId` parameter and no scoping check at all — `teams-repo.update` runs an unconditional `UPDATE ... WHERE id = teamId`. `bcs/identity-access/CONTRACT.md` already (incorrectly) documented this function as enforcing "same-organization invariants regardless of caller." Before RLS, this was a real, exploitable gap for any caller with a cross-org `teamId`; after RLS, the UPDATE would have silently matched zero rows instead — a *silent no-op*, not a thrown denial, breaking this bounded context's own established "denial is observable, not silent" convention (unlike `getUser`/`getTeamChain`/every mutation elsewhere, which all throw). Fixed identically to those: `updateTeam` now takes a mandatory `organizationId`, looks up via `teams-repo.findByOrgAndId` first, and throws for a cross-org or nonexistent `teamId`. Its one real caller (`provision-team-and-admin.ts`) was checked and did *not* need updating — closer inspection showed it imports `teams-repo.update` directly under a same-named local alias (`import { update as updateTeam }`), not the application-layer function, so it was never affected by this fix or gap in the first place; a same-name grep for call sites had initially and incorrectly flagged it.

Everything else re-checked against this final pass matches what §1–§3, §6a, and §6b above already found: `insertTeamBetween`'s unscoped `findById(tx, childTeamId)` lookup is now automatically caught earlier and more cleanly by RLS itself (a cross-org `childTeamId` returns `undefined` under the caller's `withTenantContext` scope, so `insertTeamBetween` fails fast with its existing "no team found" check, before creating any stray team row — an improvement RLS provided for free, not a gap); `listSubTeams` was already correctly scoped via `teams-repo.findByParent`. No other repo function or application-layer caller was found lacking a filter beyond what's already documented as a deliberate exception.

## 6. Denial semantics (FR-008) — confirmed, not newly decided

Re-confirmed against real code (not just the spec's assumption) that this bounded context already, uniformly, collapses "resource exists in another org" into the exact same outcome as "resource doesn't exist at all": `revoke-api-key.ts:21-26` (`ApiKeyNotFoundError` for both), `update-user.ts:35-37` (`CrossOrgUserAccessError` for both, despite its name), `authorize-invitation-management.ts:18` (`InvalidTeamAssignmentError` for both). No code change needed here — `getUser`/`getTeamChain`'s new scoped paths (§3) follow the same, already-established convention.
