# Research: Docker Compose Dev Environment

## Decision 1: Two-service layout — `app` (root Dockerfile) replaces `skillcanon` + `frontend`

**Decision**: `docker-compose.yaml` defines exactly two services, `app` and `database`. `app` builds from the repository-root `Dockerfile` (already authored in `004-ci-pipeline` for CI's `docker-build` job) rather than `legacy/backend/Dockerfile` or `legacy/frontend/Dockerfile`. Both legacy Dockerfiles are deleted once `app` replaces them (FR-008).

**Rationale**: The root `Dockerfile` already produces a single runnable image for the unified Next.js scaffold (multi-stage: pnpm install → `next build` with `output: "standalone"` → minimal runtime `node server.js`), and CI already builds it on every PR — reusing it here rather than authoring a second, compose-specific Dockerfile keeps exactly one image definition for the app, matching `004-ci-pipeline`'s stated goal of "the same image used for both self-hosted distribution and the AWS SaaS deployment."

**Alternatives considered**: Keeping three services and just repointing `skillcanon`'s build path at the new scaffold while leaving `frontend` as a legacy shell — rejected, the backlog item and its acceptance criteria explicitly call for a two-service replacement, and the new scaffold is a single unified app with no separate frontend process.

## Decision 2: Drop the legacy schema init script; Drizzle migrations own schema creation

**Decision** (from `/speckit-clarify`, integrated into spec.md's Clarifications): `database/init/001_schema.sql` is deleted. `database/init/` keeps only a `.gitkeep` so the directory still exists for `database/Dockerfile`'s `COPY init/ /docker-entrypoint-initdb.d/` step (an empty `docker-entrypoint-initdb.d/` is a normal, supported Postgres image state — the container just skips running any init SQL). `database/Dockerfile` itself is otherwise unchanged — its OpenShift-compatible non-root UID/GID hardening (`PGDATA` subdirectory, `chown`/`chmod` on `/var/lib/postgresql/data` and `/docker-entrypoint-initdb.d`, `USER 999`) carries forward as-is (FR-003).

**Rationale**: `001_schema.sql`'s own header comment says it's "Generated from alembic migrations 001 + 002 + 003" — the legacy Python backend's schema (flat `teams`/`users` tables, no per-bounded-context Postgres schemas). The new app's schema lives in `src/shared/db/schemas.ts` (seven `pgSchema()` namespaces: `identity_access`, `governance`, `prompt_registry`, `workflow`, `billing`, `audit`, `distribution`) plus per-BC `infrastructure/schema.ts` files, applied via `pnpm db:migrate` (`drizzle-kit migrate`). Running the legacy SQL against the new app's database would create tables the new app never queries, and would conflict with `context/database-conventions.md`'s ownership model. Dropping it and letting Drizzle own schema creation from an empty database is the only choice consistent with the actual code that exists today.

**Alternatives considered**: Keeping `001_schema.sql` "for now" alongside Drizzle migrations — rejected in `/speckit-clarify`: it would silently create dead, unused legacy tables in every fresh local database, actively misleading anyone inspecting the schema.

## Decision 3: No Postgres role separation introduced by this feature

**Decision**: The `database` service keeps a single Postgres role for local dev (matching today's compose-internal `POSTGRES_USER: skillcanon` / `POSTGRES_PASSWORD: skillcanon` / `POSTGRES_DB: skillcanon`), used for both the app's runtime connection and `drizzle-kit migrate`'s connection. This feature does **not** add a second, least-privileged app role or wire up `ALTER DEFAULT PRIVILEGES`/RLS grants.

**Rationale**: `context/database-conventions.md` and the root `.env.example`'s existing comments establish a two-role model (a least-privileged runtime app role for `DATABASE_URL`, a schema-owning migration role for `MIGRATION_DATABASE_URL`, never the same role) as the target production/RLS pattern. But no bounded context has a real, RLS-protected table yet — `schemas.ts` only defines the seven namespace-level schemas, with zero tables inside them. Building genuine role separation with correct privilege propagation (which requires either post-migration `GRANT`/`ALTER DEFAULT PRIVILEGES` scripting or a schema-aware bootstrap step) is a security-significant task with nothing yet to protect, and belongs with whichever future item first adds an RLS-protected table — not with this compose-wiring feature, whose backlog item names only `001-nextjs-app-scaffolding` and `002-drizzle-shared-db-kernel` as dependencies. The existing `DATABASE_URL`/`MIGRATION_DATABASE_URL` env var *names* are already correct and don't need to change later when real separation lands — only their local values will.

**Alternatives considered**: Implementing the full two-role split now "to do it right the first time" — rejected as disproportionate scope creep for a ticket about making `docker compose up -d` boot the right services; flagged here explicitly so it isn't silently forgotten (tasks.md does not include it, and a future RLS-focused backlog item should reference this decision).

## Decision 4: Compose supplies its own internal env values; `.env`/`.env.example` stays the non-Docker interface

**Decision**: `docker-compose.yaml`'s `app` and `database` services continue setting their connection values directly in the compose file's own `environment:` blocks (service-hostname-based, e.g. `postgresql://skillcanon:skillcanon@database:5432/skillcanon`), exactly as today's `docker-compose.yaml` already hardcodes `DATABASE_URL`/`AUTH_TOKEN` rather than reading them from `.env`. The root `.env.example` (already documenting `DATABASE_URL`/`MIGRATION_DATABASE_URL` with `REPLACE_ME` placeholders, from `002-drizzle-shared-db-kernel`) remains the interface for running the app **outside** Docker (`pnpm dev` / `pnpm db:migrate` against a developer's own Postgres, including one exposed by this same compose file on `localhost:5432`).

**Rationale**: This is the resolved scope from `/speckit-clarify`'s second question — `.env.example` only needs to document variables the app actually consumes today, and it already does (`DATABASE_URL`, `MIGRATION_DATABASE_URL`). Requiring a filled-in `.env` before `docker compose up -d` works would work against SC-001/SC-003's "zero manual steps beyond `docker compose up -d`" goal; keeping compose self-contained (as it already is today) avoids that requirement entirely, matching the existing precedent in this exact file.

**Alternatives considered**: Switching `docker-compose.yaml` to read `env_file: .env` — rejected: it would newly require every developer to create and fill in `.env` just to run `docker compose up -d`, a regression against today's zero-config compose boot and against SC-001/SC-003.

**Partially remediated during `as-security-scan`** (see `/speckit-analyze` finding C1): CLAUDE.md's own command table labels `docker compose up -d` "Rebuild (**self-hosted stack**)" — this file is not local-dev-only, it's also the self-host deployment mechanism. Hardcoding a known, version-controlled Postgres credential (`skillcanon`/`skillcanon`) in it is in tension with constitution Principle VI ("no security-critical setting may ship with a functional default in production"). Rather than leave this as a purely-documented, deferred gap, `POSTGRES_USER`/`POSTGRES_PASSWORD`/`POSTGRES_DB` (and the `DATABASE_URL`/`MIGRATION_DATABASE_URL` values built from them) now use Compose's `${VAR:-default}` interpolation — `skillcanon`/`skillcanon`/`skillcanon` remains the zero-config fallback (so local dev and SC-001/SC-003 are unaffected), but a self-host operator can now set `POSTGRES_PASSWORD` (and the other two) via a shell env var or a `.env` file next to `docker-compose.yaml` to get a real, non-default credential with no compose-file edit required. Verified both paths locally: default boot works unchanged, and `POSTGRES_USER=customuser POSTGRES_PASSWORD=strongpw123 POSTGRES_DB=customdb docker compose up -d` boots with the override flowing through the healthcheck, the `app` service's `DATABASE_URL`, and a live connectivity check. This is a real mitigation, not a full fix — nothing forces an operator to actually set a strong password, so a genuinely enforced (fail-if-still-default) posture is still a distinct, future security-hardening task, disproportionate to add to this ticket.

## Decision 5: Postgres readiness — compose healthcheck + `depends_on: condition: service_healthy`

**Decision**: The `database` service gets a `healthcheck:` using `pg_isready`; the `app` service's `depends_on` on `database` uses `condition: service_healthy` rather than the current unconditional `depends_on` (which only waits for container start, not Postgres readiness).

**Rationale**: Directly resolves the spec's edge case ("What happens when the app service starts before Postgres has finished initializing? The app must not crash-loop or require a manual restart") and FR-005 ("connects successfully... without any manual step") using a standard, no-new-code Compose feature — no application-level retry/backoff logic needs to be written.

**Alternatives considered**: Relying on the app's own DB client to retry on failure — rejected: `src/shared/db/client.ts`'s `getConnectionString()` fails fast and loud by design (Principle VI), and the app currently makes no DB call on `/` at all (see Decision 7), so there's no natural retry point to add one without inventing new application code the feature doesn't otherwise need.

## Decision 6: Fix now-stale doc references in README.md and CLAUDE.md

**Decision**: Update README.md's banner sentence ("`docker compose up -d` still builds and runs them [legacy backend/frontend]") and CLAUDE.md's two Notes bullets that assert `docker compose up -d` "still only builds/runs legacy/backend, legacy/frontend, and Postgres" — both become false the moment this feature merges. CLAUDE.md's own "Rebuild" table row (`docker compose up -d`) needs no edit (FR-009) since the command text itself doesn't change, only what it runs.

**Rationale**: CLAUDE.md's own established convention (documented for the `backend/`→`legacy/backend/` move) is that moving what a command targets doesn't update these files automatically — "check `docker-compose.yaml` build paths... and `CLAUDE.md`/README.md for stale references." This feature is exactly that kind of change (repointing `docker compose up -d`'s target), so the same doc-accuracy sweep applies. Left uncorrected, CLAUDE.md would actively mislead a future session (including the `as-finish` note about skipping the Docker-rebuild step "unless the branch's diff actually touches... `docker-compose.yaml`" — this branch's diff does, so that skip-condition no longer applies here and the note describing it needs to say so).

**Alternatives considered**: Leaving README/CLAUDE.md untouched and treating doc accuracy as out of scope — rejected: shipping a change that makes the project's own instructions-to-future-agents false is the exact failure mode CLAUDE.md's relocation note was written to prevent.

## Decision 7: No new application code (health endpoint) added to prove DB connectivity

**Decision**: DB connectivity is verified via the existing `pnpm db:migrate` step (FR-007/US3), not by adding a new `/api/health`-style route that queries Postgres. `src/app/page.tsx` stays a static page with no DB call.

**Rationale**: `pnpm db:migrate` already opens a real connection using `MIGRATION_DATABASE_URL` and fails loudly on any connectivity problem (`getConnectionString()`), which is sufficient to demonstrate "the app successfully connects to Postgres" end-to-end without inventing new production code whose only purpose would be to satisfy this feature's own verification step — consistent with the project's "don't add features beyond what the task requires" convention.

**Alternatives considered**: Adding a lightweight health-check route — rejected for now as unnecessary application surface for a Compose-wiring feature; nothing stops a later feature (e.g. an actual production readiness/liveness-probe need) from adding one on its own merits.
