# SkillCanon

Self-hosted prompt registry distributed via MCP, with hierarchical team/policy/objective governance. A unified Next.js/TypeScript application (pnpm-managed, `src/`) is being built at the repository root per `context/repo-structure.md`, replacing the previous split Python/FastAPI + Next.js layout; the still-functional previous implementation is preserved under `legacy/backend/` and `legacy/frontend/` while later epics port its behavior over. Postgres, deployed via Docker Compose (local) or Helm (Kubernetes).

## Key commands

| Command | Run |
|---------|-----|
| Install | `pnpm install` |
| Dev | `pnpm dev` |
| Build | `pnpm build` |
| Lint | `pnpm lint` |
| Type check | `pnpm typecheck` |
| Test | `pnpm test` |
| Rebuild (self-hosted stack) | `docker compose up -d` |

## Notes

- The commands above run against the new root-level scaffold, which currently has no business logic (empty bounded-context barrels â€” see `context/repo-structure.md`). Real functionality is ported over epic by epic from `legacy/backend/` and `legacy/frontend/`.
- `docker compose up -d` builds and runs the new unified scaffold (`app` service, from the root `Dockerfile`) plus Postgres (`database` service) â€” see epic `001-typescript-refactor-foundation`'s `005-docker-compose-dev-environment`. It no longer builds or runs `legacy/backend/`/`legacy/frontend/`; run those manually (see README.md's Quickstart) if you need the fully-functional legacy app during the transition. The database container no longer bundles a pre-initialized schema â€” `pnpm db:migrate` (Drizzle) owns schema creation from an empty database.
- Legacy backend has no type checker configured (no mypy/pyright); it was already slated for the TypeScript rewrite now underway.
- Legacy frontend has no test suite configured.
- Compliance scope: SOC2, plus NIST alignment (no dedicated automated check yet â€” review access control, audit logging, and encryption manually).
- Legacy backend tests: `cd legacy/backend && uv run pytest tests/ -v` (use `uv run`, not bare `python -m pytest` â€” deps live in uv's managed venv). If pytest/pytest-asyncio are missing, run `uv sync --extra dev` first â€” bare `uv sync` only installs the base dependency group, not the `dev` optional-dependency group.
- ESLint config uses the new flat-config format (`eslint.config.mjs`) importing `eslint-config-next`'s default export directly â€” **do not** use the `FlatCompat` shim with the legacy `"next/core-web-vitals"`/`"next/typescript"` string names; `eslint-config-next@16` already ships a native flat array and wrapping it in `FlatCompat` causes a circular-JSON crash.
- Pin `eslint@^9` and `typescript@^5` explicitly in the new scaffold, not `"latest"` â€” `eslint-config-next`'s `typescript-eslint`/plugin peer deps don't yet support `eslint@10`/`typescript@7`.
- When relocating a top-level directory in this repo (as happened moving `backend/`â†’`legacy/backend/`, `frontend/`â†’`legacy/frontend/`), check `docker-compose.yaml` build paths, `.gitignore`/`.dockerignore`, and `CLAUDE.md`/`.claude/anchorstack/project.md`/`README.md` for stale references â€” none of these update themselves.
- **Every mutating REST route must depend on `get_current_user` (or `require_admin` for admin-only ops).** `apikeys.py` and `prompts.py` were previously missing this entirely (fixed 2026-07-21) â€” when adding a new router, follow the pattern in `teams.py`/`objectives.py`/`policies.py`, and add ownership checks (self-or-admin) for user-scoped resources like API keys.
- Tests needing to exercise *real* auth (not the mocked-admin `client` fixture in `conftest.py`) should add a local `auth_client` fixture with no dependency overrides for `get_current_user`/`require_admin` â€” see `test_auth.py`, `test_prompts.py`, or `test_apikeys.py` for the pattern.
- `drizzle-kit`'s `pushSchema()` (from `drizzle-kit/api`) calls `process.exit()` internally when stdout/stdin isn't a TTY â€” it's built for the interactive `drizzle-kit push` CLI, not for programmatic use inside a test runner (kills the vitest worker outright). For Testcontainers-backed schema/column tests, use the non-interactive pair `generateDrizzleJson(imports, prevId, schemaFilters)` + `generateMigration(prev, cur)` (also from `drizzle-kit/api`) to get plain SQL statements, then execute them directly â€” see `src/shared/db/columns.test.ts`.
- RLS session context is set via `select set_config('app.current_org_id', $1, true)`, never a literal `SET LOCAL ... = $1` â€” Postgres's `SET` command doesn't accept bind parameters. See `context/database-conventions.md` and `src/shared/db/tenant-context.ts`.
- Completed backlog items get moved into `backlog/<epic>/archive/` (not just `status: done` flipped in place) â€” matches the convention already used for `backlog/000-foundations/archive/`. If an item's own Acceptance Criteria aren't all met yet (e.g. one is blocked on a separate not-yet-built backlog item), leave `status: open` and don't archive it â€” check off only what's actually true, don't force-complete.
- `eslint-plugin-boundaries` (added in epic 001's `003-module-boundary-lint-enforcement`) has real, non-obvious v7 gotchas â€” verify against the installed version's actual behavior (not just docs) before trusting a config:
  - `boundaries/entry-point` and `boundaries/element-types` are **deprecated** aliases in v7; use the canonical `boundaries/dependencies` rule with a `policies` array instead.
  - Element patterns must **not** include a trailing `**` (e.g. `src/bcs/*`, not `src/bcs/*/**`) â€” `partialMatch` (default `true`) already expands folder patterns internally, and an explicit `**` double-appends a wildcard, shifting the matched "element path" down into each subdirectory and breaking same-context (`dependency.relationship.to === "internal"`) detection.
  - The rule silently skips analyzing imports from any file that is *both* `element.isUnknown` and `file.isUnknown` â€” every top-level source directory that should be enforced (e.g. `src/app/`) needs its own `boundaries/elements` entry, even if no policy ever references it directly by type.
  - Import target resolution is real filesystem resolution (`eslint-import-resolver-node`); a nonexistent/purely-virtual import target is silently unclassified and produces zero diagnostics. Fixture tests need a real (even if transient, written-then-deleted) target file on disk â€” only the *importing* file's content can be pure in-memory text via `ESLint.lintText()`'s `filePath` option.
  - `ESLINT_PLUGIN_BOUNDARIES_DEBUG=1` prints the plugin's exact entity/dependency classification per import â€” the fastest way to debug an unexpected pass/fail.
- Speckit's `/speckit-specify` does not create a git branch in this repo (no `hooks.before_specify` registered in `.specify/extensions.yml`) â€” implementation work from `/speckit-implement` lands directly on whatever branch is checked out. Create the feature branch manually before `/speckit-implement` (or at latest before `as-finish`/`as-sync`) rather than letting substantial work accumulate uncommitted on `main`.
- `as-finish`'s Docker-rebuild step is no longer a no-op as of `005-docker-compose-dev-environment`: `docker compose up -d` now builds/runs the new scaffold (`app`) and Postgres (`database`), not the legacy stack â€” run it whenever a branch's diff touches `Dockerfile`, `database/`, or `docker-compose.yaml`.
- The repo now has **two** Dockerfiles â€” `legacy/backend/Dockerfile` and `legacy/frontend/Dockerfile` were removed in `005-docker-compose-dev-environment` once the root one covered both: root `Dockerfile` (new scaffold, added in `004-ci-pipeline`, built via `docker build .` from repo root, and now also what `docker-compose.yaml`'s `app` service builds) and `database/Dockerfile` (unchanged, used by both `docker-compose.yaml`'s `database` service and CI). The root one is also what `.github/workflows/ci.yml`'s `docker-build` job and `docker-publish.yml` build/push.
- The new scaffold has no `public/` directory yet (no static assets). The root `Dockerfile`'s runtime stage needs one for `COPY --from=build /app/public ./public` to succeed alongside `output: "standalone"` â€” the build stage runs `mkdir -p public` before `pnpm build` rather than assuming the folder exists. Remove that workaround once a real `public/` directory lands.
- CI/CD (epic 001's `004-ci-pipeline`) established the registry convention: the app image publishes to `ghcr.io/mlopstapus/skillcanon` (build-only on every PR via `ci.yml`'s `docker-build` job, push only on merge to `main` via the separate `docker-publish.yml`) â€” same registry/org as the existing Helm chart publish (`ghcr.io/mlopstapus/charts`). AWS SaaS pulling from ECR instead (per `context/deployment.md`) is an explicitly deferred, unresolved decision for a future deployment-epic item, not something `004-ci-pipeline` wired up.
- `main`'s required branch-protection check is (or will be) the single aggregate job `ci-gate` in `.github/workflows/ci.yml` (`needs:` every constituent job) â€” per `specs/004-ci-pipeline/contracts/required-check.md`, always add new CI jobs to `ci-gate`'s `needs:` list rather than registering a second required check in GitHub's branch protection settings.
- Testcontainers-backed tests (`src/shared/db/*.test.ts`) self-provision their own ephemeral Postgres container per test file via `startTestDb()` â€” a CI `test` job needs nothing beyond the runner having Docker available (true by default on GitHub's `ubuntu-latest`); don't add a redundant `services: postgres:` block to a workflow file, it would duplicate isolation Testcontainers already provides.
- The root `Dockerfile`'s runtime stage is Next.js's `.next/standalone` output, which only bundles the dependencies actually traced from the app's own import graph â€” it does **not** include the full `node_modules` (confirmed: `docker compose exec app ls node_modules` shows only `next`/`react`/`react-dom` today). A package like `postgres` or `drizzle-orm` that no route/component imports will be missing at runtime even though it's a real `dependencies` entry in `package.json` â€” `require('postgres')` inside the running container throws `MODULE_NOT_FOUND`. For an ad hoc connectivity check from inside the container, use a built-in module instead (e.g. Node's `net.createConnection`), not a driver package that isn't actually bundled.
- `docker-compose.yaml`'s Postgres credentials (`POSTGRES_USER`/`POSTGRES_PASSWORD`/`POSTGRES_DB`, and the `DATABASE_URL`/`MIGRATION_DATABASE_URL` built from them) use `${VAR:-skillcanon}`-style Compose interpolation, not a bare literal â€” this file is also the self-host deployment mechanism (see the "Rebuild (self-hosted stack)" row above), so a self-host operator can set a real credential via a shell env var or a `.env` next to the compose file with zero file edit, while local dev keeps the zero-config `skillcanon`/`skillcanon` default. Keep this pattern for any future compose-level credential; a bare hardcoded literal in this specific file is a real (if modest) security finding, not just a style nit, since it ships in a file some deployments run as-is.
- When a backlog item is archived (moved into `backlog/<epic>/archive/`), also update the epic's own `EPIC.md` Features-list link to point at `archive/<file>`, not just tick the checkbox â€” items `001`â€“`004` in `backlog/001-typescript-refactor-foundation/EPIC.md` had this exact bug (checked off but still linking to the pre-archive path) for multiple prior sessions before it was caught and fixed in `005-docker-compose-dev-environment`.
- `vitest.config.ts` needs its own `resolve.alias` mirroring `tsconfig.json`'s `paths` (`@/bcs/*`, `@/shared/*`, `@/*`) â€” Next.js resolves those natively via webpack/Turbopack, but Vitest is a separate (Vite) bundler that does not pick up tsconfig `paths` on its own; a test importing `@/shared/...` fails with "Cannot find package" until the alias is added.
- `pnpm db:generate` needs `MIGRATION_DATABASE_URL` set to a real-looking (non-`REPLACE_ME`) value even when no live DB is reachable â€” `drizzle.config.ts` evaluates `getConnectionString()` eagerly to build `dbCredentials.url`, before drizzle-kit's actual (connectionless) schema diffing runs. For local one-off generation: `MIGRATION_DATABASE_URL="postgresql://x:x@localhost:5432/skillcanon" pnpm db:generate`.
- `drizzle-kit generate` names new migrations with a random adjective-noun tag (e.g. `0001_striped_phantom_reporter`) â€” always rename the `.sql` file **and** update its `tag` in `drizzle/migrations/meta/_journal.json` to match this repo's `<timestamp>_<bc-name>_<short-description>` convention (`context/database-conventions.md`) before committing; the numbered `meta/000N_snapshot.json` files don't need renaming, only the journal's `tag` string and the `.sql` filename.
- `pnpm test -- <path>` does **not** scope the Vitest run to `<path>` â€” the `--` passthrough gets swallowed and the whole suite runs anyway. To scope a run, invoke Vitest directly: `pnpm vitest run <path>`.
- drizzle-orm's postgres-js driver wraps the real Postgres error (the one carrying the SQLSTATE `.code`, e.g. `23505` for a unique violation) inside `.cause` on a thrown `DrizzleQueryError` â€” code that maps DB errors to domain errors (e.g. a friendly "already exists" message) must check `err.cause?.code`, not just `err.code`, or the mapping silently never matches.
- When adding a Postgres advisory lock to guard a *mode-conditional* invariant (e.g. a self-hosted-only single-row guard), scope the lock acquisition to that mode too â€” an unconditional lock serializes every caller globally (including callers in the other mode, e.g. SaaS, that have no invariant to protect and would otherwise create many rows concurrently without issue).
- Any test file that creates more than one `identity_access.organizations` row (e.g. to test org-scoped behavior in another BC feature like Team Hierarchy) must `vi.stubEnv("STRIPE_ENABLED", "true")` in a `beforeEach`/`afterEach` pair â€” `createOrganization`'s self-hosted single-org guard defaults on and rejects the second `createOrganization` call otherwise. Don't re-test the guard itself in these files (already covered by `create-organization.test.ts`); just stub past it.
- A Drizzle repo function that takes `organizationId` as a parameter but doesn't actually use it in the query's `where` clause is a real cross-tenant bug, not just unused-parameter noise â€” caught once already (`teams-repo.ts`'s `findByParent` accepted `organizationId` but only filtered by `parentTeamId`, so a root-level listing query would have returned every organization's root teams). Audit every new repo function for this specifically; TypeScript won't catch it since the parameter is still "used" as a type, just not in the actual filter.
- A self-referential Drizzle FK (e.g. `teams.parent_team_id` pointing back at `teams.id`) needs an explicit `(): AnyPgColumn => teams.id` arrow-function reference (from `drizzle-orm/pg-core`), not a direct `teams.id` reference â€” the table constant isn't fully defined yet at the point the column builder runs, so a direct reference is a temporal-dead-zone error; the lazy arrow function defers evaluation until the table is complete.
- Postgres advisory locks scoped to something *other* than a single global invariant (e.g. per-organization, not per-install) should hash the scoping value rather than lock on a single shared constant â€” `pg_advisory_xact_lock(hashtext('<namespace>'), hashtext(<scopeValue>))`'s two-int32 form combines a fixed namespace (avoiding collisions with unrelated advisory locks elsewhere in the codebase) with a per-scope hash, so only operations sharing that scope value actually serialize against each other.
- Test fixtures across different `it()` blocks in the same file need unique slugs/identifiers (e.g. `` `backend-${randomUUID()}` ``, not a bare literal) whenever the assertion queries by that value without also scoping by organization/parent â€” otherwise a later test's assertion can silently match a same-named row a different, unrelated test already created. Caught twice in this repo's history (see `create-organization.test.ts`'s slug tests, and `insert-team-between.test.ts`'s original `"backend"` literal).
- Password hashing uses the `bcryptjs` npm package (pure JS), not native `bcrypt` â€” avoids the native-binding/`node-gyp` build risk on top of the `.next/standalone` bundling gotcha already noted above. `bcryptjs` produces standard, interoperable bcrypt hashes (same `$2a$`/`$2b$` format as the legacy Python `bcrypt` library) with zero native-compilation risk. Reach for `bcryptjs` in any future feature needing bcrypt hashing (password reset, invitations, etc.), not `bcrypt`.
- When a column's FK target table is created by a *later* feature (this repo's established pattern â€” e.g. `teams.owner_id` deferred no-FK until `identity_access.users` existed, added by `007-user-accounts-registration`), completing that FK afterward will break any existing test that used a fake/random UUID for that column (`update-team.test.ts`'s `newOwnerId = randomUUID()` did exactly this) â€” fix the fixture to create and use a real referenced row, not just the FK migration.
- The self-referential-FK lazy-arrow pattern (`(): AnyPgColumn => teams.id`, noted above) also applies to genuinely circular cross-table FKs, not just self-references â€” `teams.owner_id` references `users.id` while `users.team_id` references `teams.id`; whichever table is declared first in `schema.ts` needs the lazy arrow for its forward reference to the not-yet-defined table, and at least one side of the pair must be nullable (`owner_id` is) so the two rows can be inserted in either order and linked afterward.

- `identity_access.users.email`/`username` are unique per `(organization_id, email/username)` only, **not globally** (PDR-003) â€” a login-style flow that needs to resolve a user *from* an email, with no organization context yet (the user hasn't been identified), cannot filter by `organization_id` at all. Query across every organization instead (`users-repo.findByEmail`, no org filter) and resolve the organization from whichever candidate's password actually matches â€” see `008-jwt-session-auth`'s research.md Â§8 for the full reasoning and the failed-login audit-attribution edge cases (zero matches vs. exactly one vs. more than one same-email account across orgs).
- Despite `context/database-conventions.md`'s "FK to `identity_access.organizations.id`" framing, **no `organization_id` column anywhere in this codebase actually carries a Drizzle `.references()` FK** â€” not `teams.organization_id`, not `users.organization_id`, confirmed by grepping the generated migration SQL for `REFERENCES`. Match this real (not aspirational-doc) precedent for any new `organization_id` column rather than being the first to add an FK no sibling table has.
- `vi.useFakeTimers()` does not mix reliably with Testcontainers-backed DB tests in this repo â€” combining them in the same `it()` caused a test timeout that left fake-timer state bleeding into the *next* test in the file too (both failed, not just the one using fake timers). For testing JWT/session expiry against a real DB-backed flow, mock the lower-level verify function (e.g. `vi.spyOn(jwtModule, "verifySessionJwt").mockResolvedValueOnce(null)`) instead of advancing a fake clock; reserve `vi.useFakeTimers()` for pure-function unit tests with no DB I/O (e.g. `jwt.test.ts` itself, which has no Testcontainers dependency and works fine).
- Cross-BC test assertions (e.g. identity-access's tests verifying audit-compliance wrote a row) must not import another BC's internal `application/*` or `infrastructure/*` modules directly â€” `eslint-plugin-boundaries` blocks it even from a `.test.ts` file. Two ways around it that stay compliant: (1) `vi.spyOn` through the target BC's public barrel (`import * as auditCompliance from "@/bcs/audit-compliance"`, then `vi.spyOn(auditCompliance, "record")`) â€” ES live bindings mean this still intercepts the call even though the barrel just re-exports; (2) for reading rows back, use raw SQL via `sql\`select ... from audit.audit_events where ...\`` + `db.execute()` instead of importing the schema table object.
- Identity-access epic features consistently build only the `src/bcs/identity-access/{domain,application,infrastructure}` layer, never an actual `src/app/**` route or page â€” even when the originating backlog item's wording says "endpoint" (`004-jwt-session-auth`'s "Login endpoint" bullet was satisfied by an application-layer `login()` function, not a route). Both `007-user-accounts-registration` and `008-jwt-session-auth` made this same call, each documented in their own research.md â€” actual HTTP wiring and the `(auth)` UI pages stay owned by `007-distribution`, which explicitly depends on epic 002 finishing first. Don't build a route/page inside an epic-002 feature; it duplicates work Distribution's epic already owns and gets built before that epic's error-mapping/session-auth conventions exist in code.
- A locally-running `docker compose` Postgres volume (`skillcanon_pgdata` / `spechub_pgdata`, whichever name predates the current `docker-compose.yaml`) only runs `initdb` once, at volume creation â€” if the Compose file's `POSTGRES_USER`/`POSTGRES_DB` defaults change later (e.g. the SpecHubâ†’SkillCanon rename changed them from `spechub` to `skillcanon`), an existing volume keeps the *old* role/database name forever, and `pnpm db:migrate`/any connection using the new default fails with `role "skillcanon" does not exist`. Fix by recreating the volume (`docker compose down -v`, local dev data only, safe to drop) rather than debugging it as an app/migration bug.
- It's an established, precedented move in this repo for a feature in one epic to pull forward and complete *part* of a not-yet-started future epic's backlog item, when it has a real, immediate need the future epic would otherwise gate on (see `008-jwt-session-auth` completing `003-audit-compliance/001`'s schema/`record()`/redaction/index requirements early for login/logout audit events). Document it thoroughly on both sides: the pulling feature's plan.md Complexity Tracking, and a note on the future epic's own backlog item/EPIC.md explaining what's done vs. still open â€” leave that item's `status: open` and don't check off requirements it didn't actually deliver (e.g. a "retrofit existing mutations" bullet stays unchecked if nothing retrofitted anything).

- `shared/db/with-audit.ts`'s `withAudit(db, mutationFn, auditWriteFn)` does **not** pass `mutationFn`'s return value into `auditWriteFn` â€” both closures only share the same `tx`. This is fine for an UPDATE/DELETE audit event (the id is already known), but a CREATE operation whose audit event needs the new row's id has no way to get it from `auditWriteFn`. Fix: generate the id client-side (`randomUUID()`) *before* calling `withAudit`, and pass it into both closures â€” as an explicit `id` field on the insert and as the audit event's `resourceId`/`actorUserId`. `009-invitations` added this pattern (an optional `id?: string` passthrough on `invitations-repo.insert`, `users-repo.insert`, and `insertValidatedUser`'s params â€” all backward-compatible, existing callers omit it and the DB default still applies) as the first real (non-test) caller of `withAudit()`. Reuse this same client-generated-id pattern for any future CREATE+audit pairing through `withAudit`.
- A hard FK on a child table (e.g. `invitations.team_id â†’ teams.id`, `invitations.invited_by_id â†’ users.id`) means Postgres refuses to delete the parent row while **any** child row still references it, in any lifecycle state â€” not just "active" ones. Before writing a test for a "the referenced row was deleted out from under me" edge case, check whether the referencing column actually has a hard FK; if so (this codebase's default posture â€” hard-delete-everywhere, no `ON DELETE CASCADE`/`SET NULL` â€” per `context/database-conventions.md`), the scenario is likely unreachable, and the spec/data-model should document the *stronger* guarantee instead of describing a runtime check that can never fire (`009-invitations`'s original "team deleted before acceptance" edge case turned out to be exactly this â€” discovered only when the test setup itself failed with a `23503` foreign-key-violation before the actual assertion ran).
- `src/shared/logging` (a `pino` instance via `createLogger(env?)`/`logger`) and `src/shared/email` (`sendEmail({ to, subject, text })`, SMTP via `nodemailer` when `SMTP_HOST` is configured, else logs and returns) went from empty/nonexistent to real, first implementations in `009-invitations` â€” reuse these rather than reaching for `console.log` or a new ad hoc email path in future features. SES (the managed-SaaS email path from `context/third-party-services.md`) is still **not** implemented â€” no SaaS/AWS-credentialed deployment surface exists yet to build or verify it against; tracked in that document's own "Implementation status" section (not a backlog item, since the originating foundations item is already archived).
